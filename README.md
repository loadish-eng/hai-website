# H.A.I, LLC — Local Intelligence Infrastructure

This is a static React + Vite single-page site. It has no backend dependency
— it builds to plain HTML/CSS/JS that can be deployed to any static host or
cloud server.

## Stack

- React 18 + Vite 6
- React Router (client-side routing)
- Tailwind CSS + shadcn/ui components
- TanStack Query (available for future data fetching needs)

## Local development

```bash
npm install
npm run dev
```

The app runs at `http://localhost:5173` by default.

## Configuration

Copy `.env.example` to `.env` and adjust as needed:

```bash
cp .env.example .env
```

### Contact form

The contact form sends submissions through a Netlify Function
(`netlify/functions/contact.js`) that emails them via Gmail SMTP. It does
**not** send mail directly from the browser — browsers can't talk to mail
servers, so a small server-side function handles the actual sending.

| Variable | Where to set it | Purpose |
|---|---|---|
| `VITE_CONTACT_FORM_ENDPOINT` | `.env` (build-time) | Path the form POSTs to. Defaults to `/.netlify/functions/contact` — leave as-is unless you swap in a different backend. |
| `GMAIL_USER` | Netlify dashboard env vars (and locally in `.env` for `netlify dev`) | The Gmail address that sends the mail. |
| `GMAIL_APP_PASSWORD` | Netlify dashboard env vars (and locally in `.env`) | A Gmail **App Password** — not your normal Gmail password. |
| `CONTACT_TO_EMAIL` | Netlify dashboard env vars (optional) | Where submissions are delivered, if different from `GMAIL_USER`. |

**`GMAIL_USER`/`GMAIL_APP_PASSWORD`/`CONTACT_TO_EMAIL` are read server-side
only** (no `VITE_` prefix) — they are never bundled into the public site
code, unlike `VITE_`-prefixed variables.

**Generating a Gmail App Password:**

1. Enable 2-Step Verification on the Google account you want to send from, at https://myaccount.google.com/security
2. Go to https://myaccount.google.com/apppasswords
3. Create a new App Password (name it something like "H.A.I contact form")
4. Copy the 16-character password it generates and use it as `GMAIL_APP_PASSWORD`

The email will arrive *from* your Gmail address (Gmail doesn't allow
spoofing a visitor's address as the sender), but **Reply-To** is set to the
visitor's email — hitting Reply in your inbox goes straight to them.

The notification email subject/body shown on the form is generated by
`src/components/landing/ContactSection.jsx`.

### Testing the contact form locally

`npm run preview` only serves static files — it does **not** run the
Netlify Function, so the form will fail to find an endpoint. To test the
full flow locally (form → function → real email), use the
[Netlify CLI](https://docs.netlify.com/cli/get-started/):

```bash
npm install -g netlify-cli
netlify dev
```

This runs the Vite dev server and emulates the Netlify Function together
behind one local URL, so the form's fetch to `/.netlify/functions/contact`
works exactly as it will in production. Make sure `GMAIL_USER` and
`GMAIL_APP_PASSWORD` are set in your local `.env` file first.

## Deployment

Because this is a static SPA, it can be hosted anywhere that can serve files
over HTTP. A few common options:

### Netlify

A `netlify.toml` is included with the build command, publish directory, the
SPA redirect rule, and the functions directory already configured. Just
connect the repo in Netlify, add the `GMAIL_USER` / `GMAIL_APP_PASSWORD`
environment variables under Site settings → Environment variables, and
deploy — the contact form's serverless function ships automatically.

### Vercel / other static-site platforms

- Build command: `npm run build`
- Output directory: `dist`
- Make sure to configure a catch-all rewrite to `index.html` for client-side
  routing (most platforms, including Vercel, do this automatically for SPAs;
  `public/_redirects` is also picked up by several providers).
- **Note:** `netlify/functions/contact.js` is written specifically for
  Netlify's function runtime. If you host elsewhere, you'll need to port
  that handler to your platform's serverless function format (e.g. a Vercel
  API route, or an AWS Lambda) — the `nodemailer`/Gmail logic inside it can
  be reused as-is.

### Any cloud VPS, container host, or "any cloud server provider"

A `Dockerfile` and `nginx.conf` are included so you can build and run the
static site as a container anywhere Docker runs (a rented VPS, AWS
ECS/Fargate, Google Cloud Run, DigitalOcean App Platform, Azure Container
Apps, etc.):

```bash
docker build -t hai-consulting .
docker run -p 8080:80 hai-consulting
```

Then point your reverse proxy / load balancer at the container's port 80
(mapped to 8080 in the example above).

**Note:** the Dockerfile/nginx setup only serves the static site — it does
**not** run `netlify/functions/contact.js`, since that's Netlify's
serverless runtime. If you deploy this way, either point
`VITE_CONTACT_FORM_ENDPOINT` at your own mail-sending backend (running
alongside the container, e.g. a small Express server using the same
nodemailer logic), or remove the env var so the form falls back to a
`mailto:` link instead.

If you'd rather not use Docker, any static file server works — build the
project and copy the contents of `dist/` to the web root of your server
(Apache, plain nginx, Caddy, S3 + CloudFront, etc.), making sure unknown
routes fall back to `index.html`.

## Project structure

```
src/
  components/        Reusable UI (shadcn/ui primitives in components/ui)
  components/landing/ Landing-page sections (Navbar, Hero, Products, Contact, Footer, etc.)
  pages/              Route-level pages (currently just Home + 404)
  lib/                Small app utilities (query client, 404 page)
  assets/images/      Locally bundled illustrations used on the landing page
```

## Notes

- The placeholder illustrations in `src/assets/images/` are simple bundled
  SVGs styled to match the site's look — swap them for real product
  photography whenever you're ready.
- There is no authentication system or backend API in this project; it is a
  purely static marketing site.
